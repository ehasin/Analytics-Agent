1. TL;DR & Verdict

Headline This is a genuinely well-engineered text-to-SQL analytics agent with one of the more rigorous groundedness and hallucination-control architectures in the open-source space. The pipeline design, prompt discipline, deterministic verification layer, and injection-based eval harness are all production-quality thinking — not portfolio decoration. Since the original review (v1.6.4), prompt injection screening has been deployed as Stage 0, compliance validation has been made deterministic and promoted to a gate, and a column allowlist has been added to the execution layer. The remaining gaps are operational: no auth, no rate limiting, no cost controls, no CI/CD. Anyone cloning this for production needs to add those layers. The codebase gives them a solid foundation to build on.

3. What It Is

A multi-stage natural-language-to-SQL analytics agent running against the Olist Brazilian e-commerce dataset (~100k orders, 7 tables, 2016–2018). The user asks a business question in plain English; the agent classifies it, generates DuckDB SQL, executes it in-memory, narrates the findings, and runs a deterministic groundedness check on the narrative before returning it. The pipeline has three response modes (Retrieve, Explore, Reason) that control query scope and model tier, with automatic mode selection per turn. Three LLM backends are supported interchangeably (Anthropic Claude, Groq, Google Gemini) via a unified wrapper.

The system is designed around a specific thesis: LLM-driven analytics agents fail in characteristic, predictable ways (hallucinated numbers, fabricated column names, confident prose over data gaps). Every pipeline stage is an explicit countermeasure to one of those failure modes — conservative classification before SQL generation, a targeted SQL retry prompt, an anti-fabrication narrative prompt with numeric-fidelity rules, a retry loop on any unmatched number, and a deterministic regex-based groundedness verifier that does not depend on the model behaving well. The 50-case eval suite and 17-case guardrail injection eval are designed to stress-test those countermeasures specifically.

The primary artifact is the code_modules/ Python package — a production-grade refactor of a Colab notebook PoC. The Streamlit app is a demo frontend. The notebook is retained as the eval entry point. The target audience is engineers who want to understand how to build reliable text-to-SQL agents, and who may apply the same architecture to their own datasets and schemas.

4. Architecture Review

Request/Response Pipeline

User question │ ▼ [0] screen_for_injection (tier 0) ← NEW v1.6.5 • INJECTION_CLASSIFIER_PROMPT — classify CLEAN / INJECTION • blocks before any prompt template fires • fails open on error (legit questions always pass) │ INJECTION → decline + log; CLEAN → continue ▼ [1] interpret_turn (tier 1) • classify: standalone / continuation / pivot / correction • resolve follow-ups to standalone form • suggest mode (auto) or accept manual lock │ ▼ [2] classify_and_plan (tier 1, single call) • can_answer / cant_answer / clarifications_needed • generate DuckDB SQL query plan • surface analytical assumptions │ ▼ [3] execute (DuckDB in-memory) • guard_sql (sqlglot AST, SELECT-only, blocks system tables) • _validate_columns (column allowlist from DataFrames) ← NEW v1.6.6 • execute each query against registered DataFrames • on error → SQL_RETRY_PROMPT (targeted, not full replanning) │ ▼ [4] narrate + groundedness + compliance retry loop (up to 3 attempts) • attempt 1: NARRATIVE_PROMPT (numeric fidelity, anti-fabrication) → verify_groundedness (regex, no LLM) → check_compliance_deterministic (regex, no LLM) ← NEW v1.6.6 → both pass? done. any failure? attempt 2 • attempts 2–3: NARRATIVE_RETRY_PROMPT (unmatched tokens + compliance violation samples fed back) → verify_groundedness + check_compliance_deterministic again → pass? done. groundedness ratio > 0.20 after attempt 3? replace with error │ ▼ [5] update_summary (tier 0) • rolling session context ≤200 words

What Is Smart About It

  • Classification before SQL generation — can_answer / cant_answer / clarifications_needed runs as part of the same LLM call as planning. Merging them prevents the inconsistency where classify says "can_answer" but plan generates queries against absent tables. Conservative bias toward clarifications_needed over guessing.
  • Targeted SQL retry prompt — SQL_RETRY_PROMPT sends only schema + failing query + error. The original planning prompt is never re-sent on retry. This prevents the model from re-planning all queries with different labels, which would break downstream matching. Same principle as the narrative retry: minimal context, targeted fix.
  • Deterministic groundedness verifier — verify_groundedness in guardrails.py uses regex to extract numbers and named entities from the narrative and checks them against raw query results. No LLM call. Multiple false-positive exclusion classes (year tokens, range bounds, scale denominators, percentage tokens, snake_case entities) are explicitly handled. This is a real code-side verification layer, not a prompt instruction.
  • Split retry/block thresholds — _RETRY_THRESHOLD = 0 retries on any unmatched number (cheap, productive, catches derived arithmetic). _BLOCK_THRESHOLD = 0.20 hard-blocks only when the final attempt is still clearly bad. Low-ratio persistent noise passes rather than erroring. The design intent is explicit in the code comments.
  • Prompt injection screen as Stage 0 (v1.6.5) — screen_for_injection() runs a tier-0 LLM call on the raw question before interpret_turn or any other prompt template processes it. Designed with correct risk asymmetry: fails open (defaults to CLEAN on model error) so a guardrail failure never blocks a legitimate question. False positive = polite decline; false negative has classify_and_plan as backstop. Blocked turns logged with classification="injection_blocked".
  • Deterministic compliance gate (v1.6.6) — check_compliance_deterministic() replaces tier-0 LLM compliance scan for rules 1 (approximation language), 2 (evaluative adjectives), 4 (currency drift) with pure regex/word-list — zero LLM cost per turn, zero model-variability risk. Rules 3 and 5 dropped: 3 (motivational inference) has no reliable deterministic signal; 5 (unsupported claims) is already covered by verify_groundedness. Every rule either gates or gets dropped — log-only rules provide no stakeholder trust guarantee.
  • Column allowlist from DataFrames (v1.6.6) — execute_queries auto-builds {table: frozenset(cols)} from the DataFrames it already holds, guaranteeing sync with the actual schema. _validate_columns() walks the sqlglot AST for qualified column references; unqualified names and CTE aliases skipped to avoid false positives. build_column_allowlist(data_model) also available as public API for restricted-column annotations in data_model.json.
  • Three-tier model routing with per-call timeouts — tier 0 (cheap) for injection screen and summaries, tier 1 (default) for planning and narration, tier 2 (strong) only for Reason-mode narration. 180s ceiling on all backends including Gemini (enforced via thread future since the google-genai SDK has no per-call timeout).
  • Stage trace for full observability — every turn produces a stage_trace list with per-stage timing, tier, retry counts, grounding results, compliance results, and narrate_attempts. This is logged to the DB and available for dashboards without any additional instrumentation.
  • DOC_LOOKUP fast path — metadata questions ("what tables are available?") are answered directly from data_model.json without SQL generation. Groundedness is skipped (no data rows to ground against). Avoids false positives on schema descriptions.
  • Injection-based guardrail eval — rather than relying on the model to hallucinate naturally, the eval harness injects known-bad SQL and known-bad narratives at controlled pipeline stages, then asserts the guardrail fired. This is a meaningfully stronger test methodology than end-to-end accuracy alone.

What I Would Question

  • Column allowlist — resolved in v1.6.6 — guard_sql now validates qualified column references via _validate_columns() (sqlglot AST walk) against an allowlist auto-built from DataFrames. build_column_allowlist(data_model) exposed as public API for future restricted-column annotations in data_model.json. The Olist dataset has no sensitive columns; in real-schema deployments, annotations in data_model.json are the next step.
  • No query row or cost limits — DuckDB is in-memory on the demo, and the 7 Olist tables are small. Against a larger schema, an LLM-planned SELECT * FROM large_table would return millions of rows. The 50K char truncation cap in the narration stage limits what the LLM sees, but the full result is still computed. A per-query row limit (e.g., LIMIT 10,000 appended to all queries without explicit LIMIT) should be enforced at the execution layer.
  • Compliance is log-only — resolved in v1.6.6 — check_compliance_deterministic() replaces the tier-0 LLM compliance scan. Rules 1 (approximation language), 2 (evaluative adjectives), 4 (currency drift) are checked via regex/word-list — zero LLM cost, zero model-variability risk. Violations now gate narrative retries with rule-specific feedback fed back via {compliance_feedback} in NARRATIVE_RETRY_PROMPT. Hard-block remains groundedness-only. Rules 3 (motivational inference) and 5 (unsupported claims) are dropped — 3 has no reliable deterministic signal; 5 is covered by verify_groundedness.
  • Prompt injection screening — resolved in v1.6.5 — screen_for_injection() deployed as Stage 0 tier-0 LLM pre-classifier before interpret_turn. Blocks override instructions, system-prompt exfiltration, and identity manipulation. Fails open on error so guardrail failure never blocks a legitimate question. README §"Known limitations" updated.
  • Model version pinning — partially addressed in v1.6.5 — MODEL_MAP now has a docblock explicitly documenting that the string identifiers are intentional pins ("Never use 'latest' or unversioned aliases"), when pins were last reviewed, and the procedure for updating them. Provider-level version suffixes (e.g., claude-sonnet-4-6-20250514) are not used — not all providers expose stable version strings. The docblock discipline is the practical mitigation.
  • Colab-specific tooling in eval_runner.py — run_eval and run_guardrail_eval import IPython.display and the Google Drive API directly. These calls are try-wrapped but not cleanly separated. Someone running the eval in a non-Colab environment gets silent fallback behavior, not a clean programmatic interface.

Specific Code Wins

  • prompts.py — all prompts centralized: Every LLM call in the pipeline uses a named constant from one file. Changing a prompt rule requires one edit in one place, and the change is immediately visible to all callers. The naming convention (NARRATIVE_RETRY_PROMPT, SQL_RETRY_PROMPT) makes the retry architecture self-documenting.
  • guardrails.py — false-positive exclusion classes: Year tokens, range bounds, scale denominators, percentage tokens, and snake_case entity normalization are each handled with a specific regex and a comment explaining precisely why. This is engineering discipline, not just "it works."
  • analyst_agent.py — _RETRY_THRESHOLD / _BLOCK_THRESHOLD constants: Separating "when to retry" from "when to block" with explicit, documented constants is a design choice that prevents the two concerns from being conflated in production. The in-code calibration notes tell cloners exactly what to tune.
  • duckdb_utils.guard_sql(): Uses sqlglot AST parsing, not regex. Blocks on statement type, not keyword matching. Handles the sqlglot_parse_failed case explicitly with a warn-and-allow path rather than a silent pass. The distinction between ValueError (blocked) and RuntimeError (parse failure) is clean.
  • call_llm() error categorization: Distinguishes fatal errors (413, quota exhausted — never retry), TPM rate limits (429 — short backoff, parse wait time from error message), and transient overload (529 — exponential backoff). This is correct production LLM client behavior.
  • _async_log_turn() never re-raises: DB logging errors are printed to Streamlit Cloud logs but never propagate to the UI. Logging failure never becomes a user-facing error. The try/except is explicit about this contract.

Specific Concerns

  • streamlit_app.py at 1,534 lines: The file is large but mostly CSS (~400 lines) and JS (~100 lines). The actual Python orchestration logic is ~500 lines and is reasonably structured. Not a maintainability crisis, but splitting CSS/JS into a separate assets/ file would help. The threading poll loop (lines 1155–1345) is complex — a bug here affects every user session.
  • Stale comment in prompts.py — resolved in v1.6.5: The comment above NARRATIVE_RETRY_PROMPT referenced the old >30% threshold. Fixed in commit 143d62a to reflect _RETRY_THRESHOLD = 0 (any single unmatched number).
  • eval_runner.py Colab coupling: IPython.display and Google Drive API imports are try-wrapped but still present at module level. _gdrive_link() is 50 lines of Drive API traversal inside the log writer. This should be an optional plugin, not embedded in the core eval runner.
  • No row limit in execute_queries(): The function executes whatever SQL is planned with no row cap. The 50K char truncation in the narrate prompt is a downstream safety net, not a query-level guard. A SELECT * against a large table will still compute a full DataFrame before being truncated.
  • JavaScript setInterval polling: streamlit_app.py injects two setInterval loops (slider fix at 400ms, autocomplete fix at 600ms) that run for the entire browser session. This is a fragile approach to Streamlit CSS/JS customization that breaks on Streamlit framework updates.
  • No .env.example — resolved in v1.6.5: .env.example added with all required keys, Colab secret name mapping, and guidance for local vs. Colab vs. Streamlit deployment paths.

6. The Eval Claim

Claim "50/50 standard eval · 17/17 guardrail injection eval" — stated in README and status line.
DimensionAssessment
Sample size 50 standard cases is below the statistical noise floor for tight pass-rate claims. A single regression looks like a 2% drop (1 case). The README acknowledges this directly and correctly. The number is a meaningful signal, not a rigorous benchmark.
Dataset quality Good diversity across 6 categories: metadata (6), easy (10), hard (10), misleading (10), multi-stage (5), realistic (9). Misleading questions and multi-stage analysis are the right stress tests for an analytics agent. Most OSS evals stop at easy lookups.
Self-eval bias Present by default. run_eval() uses llm_fn as both answerer and assessor unless assessor_llm_fn is provided. The same model family grading its own answers inflates pass rates. Cross-model grading is supported via assessor_llm_fn but is not the default. The 50/50 headline was likely produced with self-grading.
Validation methodology Dual validation (lambda + LLM) with reconciliation. The lambda is the primary signal; LLM assessment is logged for human review and can override in either direction via conservative thresholds. Lambdas must be written to match the agent's actual output format — brittle string matching (e.g. month abbreviations when the agent outputs ISO dates) produces false failures that the reconciliation cannot reliably recover from. Lambda robustness is the critical maintenance surface.
Variance / repeated runs Not reported. LLM outputs are non-deterministic. The 50/50 result is from a single run. A 10× loop on a subset (easy set) is mentioned in the changelog context. Full suite variance across runs is not measured or documented.
Model version pinning Not pinned. The eval results reflect the model state at time of last run. A provider update can change pass rates without any code change.
Guardrail injection eval 17 cases designed to test specific failure modes via injection rather than relying on natural hallucination. SQL injection (5), narrative injection (9), prompt injection (3). False-positive checks included (GQ11, GQ13, GQ14, GQ17). This methodology is stronger than end-to-end accuracy alone — it tests the guardrail components directly and in isolation.
What the headline means "50/50" means: on one run, with self-grading, against 50 questions from a known dataset, the agent produced answers that satisfied both a deterministic lambda and an LLM assessor. It does not mean: reliably correct on unseen business questions across model updates. The README is accurate about this. The headline reads stronger than the evidence in isolation; the documentation corrects for it.

8. Trust Issues & Risks

High1. No rate limiting or cost controls on public demo
Evidence
README: "Any visitor can trigger unlimited Reason-mode queries (the most expensive tier) against the owner's API key." Confirmed in streamlit_app.py — no per-session token budget, no daily cap, no tier-gating.
Impact
A single session running 50 Reason-mode turns can cost $10–30 against the Opus-class model. No billing protection exists.
Mitigation
Reason-mode behind auth; per-session token budget in call_llm; provider billing alerts.
Low2. Prompt injection — addressed in v1.6.5
Status
Resolved. screen_for_injection() deployed as Stage 0 tier-0 LLM pre-classifier. Runs on the raw question before any prompt template fires. Fails open on model error (legit questions never blocked by a guardrail failure). Blocked turns logged with classification="injection_blocked".
Residual risk
Tier-0 model errors fail open by design. Adversarial prompts outside the classifier's training distribution may pass to classify_and_plan (which is itself a defense layer). Input length is uncapped. Acceptable posture for supervised internal use; public service requires additional IP-level rate limiting.
Low3. Column allowlist — addressed in v1.6.6
Status
Resolved. _validate_columns() added: sqlglot AST walk checking qualified column references against an allowlist auto-built from DataFrames in execute_queries. Always in sync with actual schema. guard_sql accepts optional column_allowlist parameter; build_column_allowlist(data_model) exposed as public API.
Residual risk
Current implementation allows all registered schema columns (appropriate for the public Olist dataset). For real-data deployments with PII or sensitive columns: annotate restricted columns in data_model.json and pass a narrowed allowlist. The infrastructure is now in place; the schema annotations are the next step.
Medium4. Model version drift — eval results become invalid after provider updates
Evidence
MODEL_MAP in llm_backends.py references model names ("claude-sonnet-4-6", "llama-3.3-70b-versatile") without version pinning. Provider aliases are mutable — the same name can point to different checkpoints after a provider update.
Impact
Prompt behavior, eval pass rates, threshold calibration, and classification accuracy can all change silently after a provider model update. Production systems cloning this codebase would not know when this happened.
Mitigation
Pin specific model versions where available; log the model version used in stage_trace per turn; re-run eval after any model update.
Medium5. DB logging records narratives and full output — PII risk in real-data deployments
Evidence
_async_log_turn in streamlit_app.py calls logger.log_turn(narrative=result.get("narrative_raw", ""), raw_output={...}). Full narrative text and stage_trace (which can include query result excerpts) are written to the Neon DB. Acceptable for the Olist public dataset.
Impact
In a deployment against real business data, narratives that surface revenue figures, customer counts, or operational metrics get logged to a third-party database (Neon) without redaction.
Mitigation
Add a configurable narrative redaction flag; document this gap explicitly in the README for cloners.
Low6. Compliance validation — made deterministic gate in v1.6.6
Status
Resolved. check_compliance_deterministic() replaces tier-0 LLM compliance call. Rules 1 (approximation language), 2 (evaluative adjectives), 4 (currency drift) checked via regex/word-list — zero LLM cost, zero model variability. Compliance violations now trigger narrative retries (same as numeric unmatched). Rules 3 (motivational inference) and 5 (unsupported claims) dropped: 3 has no reliable deterministic signal; 5 is covered by verify_groundedness.
Design note
Compliance retries but never hard-blocks — a sentence can be rewritten; a fabricated number cannot. Violation samples carry forward into NARRATIVE_RETRY_PROMPT via {compliance_feedback} slot with rule-specific correction advice. Rule 3 (motivational inference) is an accepted false negative — the narrative prompt (tier-1 Sonnet) remains the enforcement layer.
Low7. No query row limit — large schema queries compute full DataFrames
Evidence
execute_queries() in duckdb_utils.py executes SQL as-is. The 50K char narration cap truncates what the LLM sees, but the full DataFrame is computed first. On the 100k-row Olist dataset this is fast. Against larger tables, this becomes a latency and memory issue.
Impact
Slow queries for cloners with larger datasets; potential memory pressure in serverless deployments.
Mitigation
Add an optional row_limit parameter to execute_queries; append LIMIT N to unbounded SELECTs at the guard layer.

This system takes hallucination seriously as a verification problem, not just a behavior problem. The protection architecture is:

11. Strengths Summary

  • The deterministic groundedness verifier is the most distinctive feature. verify_groundedness() is a real code-side verification layer that does not depend on the model behaving well. The false-positive exclusion classes (percentages, year tokens, range bounds, scale denominators, snake_case entities) are each individually motivated and documented. This is not seen in most text-to-SQL OSS projects.
  • Prompt discipline is among the strongest in open-source analytics agents. The anti-fabrication, numeric-fidelity, scope-discipline, and anti-inference rules in NARRATIVE_PROMPT reflect a sophisticated understanding of how LLM-driven analytics agents fail. The prompt is a design artifact, not boilerplate.
  • The SQL retry prompt is architecturally correct. Sending only schema + failing query + error (not the full planning prompt) on retry is the right tradeoff. It prevents label drift and token waste while fixing the specific failure. The same targeted-fix principle is applied correctly to narrative retries.
  • The pipeline is testable by design. llm_fn and execute_fn are injected callables. The guardrail eval harness injects known-bad SQL and narratives at controlled stages. This makes individual pipeline components testable in isolation, which is rare in production agent code.
  • Known limitations are documented honestly. The README explicitly lists eval sample size concerns, no cost controls, no prompt injection screening, HTTP geo-IP, and self-grading bias. This is more honest than most comparable projects and is the correct posture for a reference architecture.
  • Three-tier model routing reduces cost without sacrificing quality where it matters. Tier-0 for summaries and compliance, tier-1 for planning and narration, tier-2 only for Reason-mode narration. The routing logic and model assignments are explicitly documented.
  • The Streamlit app threading model is handled correctly. Daemon thread per turn, stop event for cancellation, session-scoped result dict, graceful hot-reload recovery. Most Streamlit apps that try to run agents in threads get this wrong.

12. Gaps & Weaknesses Summary

  • No CI/CD. The most consequential operational gap. A regression in verify_groundedness or guard_sql is not caught until the next manual eval run. For a project positioning itself as a reliability reference, having no automated test gate is a contradiction.
  • No auth, rate limiting, or cost controls on the public demo. Three separate production blockers documented in the README but not mitigated. Anyone cloning this as their starting point inherits all three gaps without any scaffolding to add them.
  • Column allowlist for restricted schemas — partially addressed (v1.6.6). Infrastructure in place: _validate_columns() validates against auto-built allowlist. Real-data deployments need restricted-column annotations in data_model.json — not yet templated.
  • Eval self-grading is the default. The 50/50 headline was produced with the same model family answering and grading. Cross-model grading is supported but not the default. The reported pass rate should be treated as an upper bound.
  • Compliance — resolved (v1.6.6). check_compliance_deterministic() replaces tier-0 LLM scan. Rules 1/2/4 gate retries. Rule 3 (motivational inference) accepted false negative — no reliable deterministic detection; narrative prompt is enforcement layer. Rule 5 covered by verify_groundedness.
  • Analytical interpretation correctness is not verified. Numeric groundedness is checked. Whether the analytical framing is correct (right metric grain, right comparison logic, appropriate use of averages vs. distributions) is not. This is the hardest gap to close deterministically, but it is where real analytics errors live.
  • Dependency versions are unpinned. A dependency upgrade between the eval run and a cloner's first deployment can produce different behavior. No lockfile.
  • Colab coupling in eval_runner.py. IPython.display and Google Drive API calls are embedded in the core eval runner. Non-Colab users get try-wrapped silent fallbacks rather than a clean interface.
  • Stale comment in prompts.py — resolved (v1.6.5). Fixed in commit 143d62a.
  • No production deployment template. The README documents the gaps but provides no scaffolding for adding auth, rate limiting, or column allowlisting. A cloner is told what to add but not given a starting point.

14. Bottom Line

The core pipeline and guardrail architecture are production-quality thinking wrapped in a portfolio-grade operational shell. The agent design — conservative classification, targeted retries, deterministic verification, injection-based eval — reflects a genuine understanding of how text-to-SQL agents fail in practice. The operational layer around it (auth, rate limiting, cost controls, CI, model pinning) is absent but not surprising for an open-source reference project at this stage.

For the stated use case — open-source reference architecture that future production systems will be built from — the most urgent work is not adding features to the pipeline. It is adding the production scaffolding that cloners will need but won't find: CI, a deployment checklist, column allowlisting, dependency pinning, and a cross-graded eval result that stands behind the headline number. The gaps are documented; the next step is to template the solutions.

Final Recommendation Deploy as reference architecture and portfolio piece. Do not deploy as a public production service without adding auth, rate limiting, cost controls, column allowlisting, model version pinning, and CI. The agent core is trustworthy. The operational shell is not production-ready. Cloners who understand that distinction — and are willing to add the missing layer — will find this codebase a solid and unusually honest starting point.
Strongest production-relevant trait worth preserving The split between _RETRY_THRESHOLD (any unmatched number triggers a retry) and _BLOCK_THRESHOLD (20% ratio triggers a hard block) is the right design for production analytics agents — it is cheap to retry early, expensive to fail the user. Preserve this architecture and extend it to the compliance layer when adding deterministic rule-based compliance checks.
